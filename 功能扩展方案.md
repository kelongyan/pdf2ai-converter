# PDF2AI Converter 功能扩展方案

> 基于 v3.0.0 代码分析，梳理可行的功能扩展方向、实现路径与优先级。

---

## 一、输出格式扩展

### 1.1 PDF → HTML 输出

**动机**：HTML 可保留比 Markdown 更丰富的样式（颜色、字体、多栏布局），且浏览器直接可预览，无需额外工具。

**实现路径**：

- 复用 Word 精确模式的 JSON 提取结构（heading/paragraph/table + style 字段）
- 新增 `models/html_generator.py`，将 JSON 渲染为语义化 HTML + 内联 CSS
- 入口：`main_html.py`，CLI 参数与 `main_word.py` 对齐
- Launcher 菜单新增 HTML 选项，`_MENU_ACTIONS` 扩展即可

**关键设计**：

```
JSON 结构 (已有)
    ↓
HtmlGenerator.generate_html(pages_data, output_path)
    ↓
单文件 .html（内联 CSS，无外部依赖）
```

**工作量估算**：~200 行新代码 + 测试

---

### 1.2 PDF → LaTeX 输出

**动机**：学术论文场景高频需求。当前 Markdown 管道已要求模型输出 LaTeX 公式，但文档结构仍是 Markdown。

**实现路径**：

- 新增 prompt 模板，要求模型直接输出 `\section{}`、`\begin{table}`、`\begin{equation}` 等结构
- 输出模式为纯文本拼接（类似 Markdown 管道），无需额外渲染库
- Cache key 中 `output_mode` 设为 `"latex"`

**注意事项**：

- LaTeX 对格式错误容忍度极低，需要更严格的质量检查
- 可考虑生成后自动调用 `pdflatex` 编译验证（可选）

**工作量估算**：~150 行新代码 + prompt 调优

---

## 二、模型层扩展

### 2.1 多模型适配器

**现状**：`models/base.py` 定义了 `VisionModel` 抽象基类，但只有 `OpenAICompatibleModel` 一个实现。`config.yaml` 中已有 `model.type` 字段但未被路由使用。

**扩展目标**：

| 适配器 | 目标平台 | 差异点 |
|--------|----------|--------|
| `OpenAICompatibleModel` | GPT-4o, 通义千问, 智谱等 | 已实现 |
| `AnthropicModel` | Claude 3.5/4 系列 | image 放在 content block 而非 image_url |
| `OllamaModel` | 本地 LLaVA, Qwen-VL 等 | 无需 API Key，base_url 固定为 localhost |
| `GoogleModel` | Gemini Pro Vision | 使用 google-generativeai SDK |

**实现路径**：

```python
# pipeline_utils.py 中的路由逻辑
def build_vision_model(config: dict) -> VisionModel:
    model_type = config["model"]["type"]
    if model_type == "openai_compatible":
        return OpenAICompatibleModel(...)
    elif model_type == "anthropic":
        return AnthropicModel(...)
    elif model_type == "ollama":
        return OllamaModel(...)
    else:
        raise ValueError(f"不支持的模型类型：{model_type}")
```

**工作量估算**：每个适配器 ~80 行，路由逻辑 ~20 行

---

### 2.2 模型回退链

**动机**：大模型 API 不稳定时（超时、限流、服务降级），自动切换到备用模型，避免整个任务失败。

**配置设计**：

```yaml
model:
  type: openai_compatible
  name: gpt-4o
  # ...
  fallback:
    name: gpt-4o-mini
    base_url: https://api.openai.com/v1  # 可选，默认复用主模型的
```

**实现路径**：

- 在 `_send_image_prompt` 的最终异常处理中，检查是否配置了 fallback
- 如果有，构造备用 client 重试一次
- 日志中标记该页使用了降级模型

**工作量估算**：~50 行改动

## 三、处理能力增强

### 3.1 图片/图表提取

**动机**：当前管道只提取文本，页面中的图表、流程图、照片被丢弃或仅以文字描述代替。

**实现路径**：

1. 在 prompt 中要求模型返回图表区域的坐标（bounding box）
2. 用 PyMuPDF 从原始 PDF 页面裁剪对应区域
3. 保存为独立图片文件（与输出文件同目录）
4. 在 Markdown/Word 中插入引用

**JSON 扩展**：

```json
{
  "type": "image",
  "description": "图 3-1 系统架构图",
  "bbox": [72, 200, 540, 450]
}
```

**挑战**：

- 模型返回的坐标精度有限，需要容错处理
- 需要将 PDF 坐标系（左下原点）转换为像素坐标

**工作量估算**：~300 行新代码

---

### 3.2 OCR 预处理层

**动机**：扫描件 PDF（纯图片无文字层）完全依赖视觉模型识别，token 消耗大且容易出错。加一层 OCR 预处理可以：
- 将 OCR 文本作为 context 传给模型，降低幻觉
- 对于简单页面，OCR 结果可能已经足够，无需调用大模型

**实现路径**：

```
PDF 页面图片
    ↓
OCR 引擎（Tesseract / PaddleOCR）
    ↓
OCR 文本作为 prompt context
    ↓
视觉模型（校正格式、补充结构）
```

**配置**：

```yaml
ocr:
  enabled: false
  engine: tesseract  # tesseract | paddleocr
  languages: ["chi_sim", "eng"]
  confidence_threshold: 0.8  # 高于此阈值的页面跳过大模型
```

**工作量估算**：~200 行 + 新依赖（pytesseract 或 paddleocr）

---

### 3.3 增量/流式输出

**动机**：大 PDF（100+ 页）处理耗时长，当前必须全部完成才写文件。中途崩溃则前功尽弃。

**实现路径**：

- Markdown 管道：每处理完一页，立即 append 到输出文件
- Word 管道：维护一个 `pages_data` 的持久化中间文件（JSON Lines），最终一次性生成 docx
- 在输出文件头部写入元数据注释，标记已完成页数

**兼容性**：与现有 cache 机制互补——cache 保证页面级幂等，流式输出保证文件级可见性。

**工作量估算**：~100 行改动

## 四、用户体验提升

### 4.1 Token 用量统计与成本估算

**动机**：用户最关心的问题之一是"这次转换花了多少钱"。当前完全没有追踪。

**实现路径**：

- 从 OpenAI API response 中提取 `usage` 字段（`prompt_tokens`, `completion_tokens`）
- 在 `_send_image_prompt` 返回时同时返回 usage 信息
- 累计统计，任务结束时打印汇总：

```
📊 API 用量统计：
   总请求数：25
   输入 Token：125,000
   输出 Token：45,000
   估算费用：$2.15（基于 GPT-4o 定价）
```

**配置**：

```yaml
stats:
  enabled: true
  pricing:  # 可选，用于费用估算
    input_per_1k: 0.005
    output_per_1k: 0.015
```

**工作量估算**：~80 行改动

---

### 4.2 Web UI 服务化

**动机**：降低使用门槛，提供拖拽上传、实时进度可视化、结果在线预览等浏览器端体验，替代当前 CLI + Tkinter 交互方式。

**技术栈**：FastAPI 后端 + React 18 + shadcn/ui + Tailwind CSS + Framer Motion 前端。

**详细方案**：参见 [Web_UI_可视化方案.md](./Web_UI_可视化方案.md)，包含完整的技术选型论证、架构设计、目录结构、API 设计、页面设计、设计系统规范、实现阶段划分等。

---

### 4.3 批量任务队列

**动机**：当前批量模式是同步串行，无法暂停/恢复，无法设置优先级。

**实现路径**：

- 任务队列持久化为 JSON 文件（`tasks.json`）
- 每个任务记录：PDF 路径、输出格式、状态（pending/running/done/failed）、进度
- 新增 CLI 命令：

```powershell
python task_manager.py add *.pdf --format markdown
python task_manager.py status
python task_manager.py run          # 开始处理队列
python task_manager.py pause
python task_manager.py retry-failed
```

**工作量估算**：~250 行新代码

## 五、质量与可靠性

### 5.1 Word 输出质量检查

**现状**：`QualityChecker` 只服务于 Markdown。Word 管道没有质量校验。

**检查项**：

| 检查项 | 说明 |
|--------|------|
| 空页面检测 | elements 为空的页面占比 |
| 失败页面统计 | content 包含失败标记的页面 |
| 元素类型分布 | 是否全是 paragraph（可能丢失了表格/标题） |
| 模型拒绝响应 | 检测"I cannot"等模式 |
| JSON 降级统计 | precise 模式降级为 fast 的页面数 |

**实现路径**：

- 新增 `WordQualityChecker` 类，输入为 `pages_data` 列表
- 在 `main_word.py` 的 `generate_word` 之前调用

**工作量估算**：~120 行新代码

---

### 5.2 输出对比工具

**动机**：换模型、调 prompt、改 DPI 后，需要对比效果差异。

**实现路径**：

```powershell
python diff_tool.py output_v1.md output_v2.md --report diff_report.html
```

- 对 Markdown：逐页 diff，高亮差异段落
- 对 Word：提取纯文本后 diff
- 输出 HTML 报告，左右对照显示

**工作量估算**：~200 行新代码

## 六、配置与安全

### 6.1 API Key 安全存储

**现状**：`.config.json` 中 API Key 仅 base64 编码，等同明文。

**方案对比**：

| 方案 | 安全性 | 跨平台 | 依赖 |
|------|--------|--------|------|
| 系统 Keyring | 高 | 是 | `keyring` 库 |
| AES 加密 + 主密码 | 中高 | 是 | `cryptography` 库 |
| 环境变量 | 中 | 是 | 无 |

**推荐**：使用 `keyring` 库，Windows 上自动使用 Credential Manager。

```python
import keyring

keyring.set_password("pdf2ai", profile_name, api_key)
api_key = keyring.get_password("pdf2ai", profile_name)
```

**回退**：如果 keyring 不可用（无桌面环境），降级为当前 base64 方式并警告。

**工作量估算**：~60 行改动 + 新依赖

---

### 6.2 配置导入/导出

**动机**：团队共享模型配置（脱敏 API Key）。

**实现**：

```powershell
# 导出（API Key 替换为占位符）
python config_manager.py export --profile default --output profile.json

# 导入（提示输入 API Key）
python config_manager.py import --file profile.json
```

**导出格式**：

```json
{
  "name": "default",
  "api_url": "https://api.openai.com/v1",
  "api_key": "***REDACTED***",
  "model": "gpt-4o",
  "dpi": 150,
  "chunk_size": 10
}
```

**工作量估算**：~50 行新代码

## 八、架构演进建议

当前架构已经具备良好的扩展性基础：

- `VisionModel` 抽象基类支持多适配器
- `pipeline_utils.py` 集中了共享逻辑
- Cache key 设计包含 output_mode，天然支持新格式
- 数据驱动的菜单分发，新增选项只需加一行

**建议的架构调整**（在 Phase 2 开始前）：

1. 将 `build_vision_model` 改为工厂模式，根据 `model.type` 路由
2. 抽象出 `OutputGenerator` 基类（Markdown/Word/HTML/LaTeX 共用接口）
3. 将质量检查抽象为 `QualityChecker` 基类 + 格式特定子类

这些调整不改变外部行为，但为后续扩展铺平道路。
