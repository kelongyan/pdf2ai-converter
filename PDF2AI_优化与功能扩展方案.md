# PDF2AI Converter 项目优化与功能扩展方案

## 总体原则

这份方案按两个标准重新排序：

1. **优先级**：是否直接影响当前主链路的稳定性、可恢复性和扩展性。
2. **性价比**：投入是否小、见效是否快、能否减少后续返工。

排序原则很直接：

- **先做低成本高收益**。
- **再做中成本高复用**。
- **最后做高成本长周期**。
- 对未来能力有强依赖的工作，必须排在依赖项之后。

一句话版本：**先收口现有系统，再搭通用骨架，最后做大功能和产品化。**

---

## 第一梯队：最高优先级、最高性价比

这部分最该先做，因为改动相对可控，但能明显提升当前系统可用性，并且直接降低后续开发返工。

### 1. 统一现有配置项与真实行为

**优先级：极高**  
**性价比：极高**

仍需处理：

- 统一 `output.add_page_separator` 等已有配置项在主流程中的实际生效逻辑。
- 检查 Markdown / Word 两条主链路对相同配置项的行为是否一致。
- 清理“配置里有字段，但运行时没有完整使用”的情况。

为什么排第一：

- 改动小。
- 影响面明确。
- 能立即减少“配置写了但没生效”的隐性 bug。
- 这是后续做 CLI 参数覆盖、批处理和产品化前必须先收口的基础。

### 2. 补齐边界测试

**优先级：极高**  
**性价比：极高**

仍需继续补：

- 配置项真实行为测试。
- 缓存恢复与部分页命中测试。
- 并发异常分支测试。
- 重试与失败页路径测试。
- 批量任务相关测试。

为什么这么靠前：

- 投入比新功能低。
- 能给后续所有改动兜底。
- 不先补测试，后面做恢复能力、AST、OCR 都容易改一处炸一片。

### 3. 增强失败恢复与重试体验

**优先级：极高**  
**性价比：高**

仍需完成：

- 失败页重试入口。
- 更明确的失败页恢复流程。
- 中断后恢复清单或恢复指引。
- 更完整的任务状态管理。

为什么现在就做：

- 这是用户最容易感知到的稳定性提升。
- 对长 PDF 和异常场景收益很大。
- 大部分能力可以复用现有缓存和失败页信息，不算从零起步。

### 4. 增强可观测性和诊断能力

**优先级：高**  
**性价比：高**

仍需补齐：

- 结构化日志。
- 请求耗时统计。
- 批量任务统计。
- 更清晰的 debug 输出。
- 更易定位的失败原因汇总。

为什么值得尽早做：

- 能明显降低排障成本。
- 后续做批量任务、并发治理、OCR 混合模式时都会依赖这些信息。
- 这类工作不炫，但非常省未来时间。

### 5. 增强批量处理能力

**优先级：高**  
**性价比：高**

仍需实现：

- 跳过已完成文件。
- 失败任务清单。
- 批量任务结果汇总。
- 中断后继续处理。
- 更清晰的批处理参数和报告能力。

为什么还在第一梯队：

- 这是把“单文件工具”变成“真能干活的工具”的关键一步。
- 大量逻辑可以建立在现有缓存、失败页、日志能力上。
- 相比 Web/API，这类能力更贴近当前项目阶段，也更便宜。

---

## 第二梯队：高优先级、高复用价值

这部分不是最先动手，但它们决定项目能不能从“双输出脚本”长成“可持续扩展的平台骨架”。

### 6. 引入统一中间表示（AST / IR）

**优先级：高**  
**性价比：中高**

仍需完成：

- 设计统一的 Page / Element AST。
- 把 Markdown 和 Word 输出逐步收敛到统一中间层。
- 解耦“识别”和“导出”两个阶段。

推荐目标结构：

```text
PDF
 ↓
Page Image
 ↓
Vision/OCR Extractor
 ↓
Page/Element AST
 ↓
Exporter
```

为什么排在这里：

- 这是未来多格式输出、OCR 混合、自动修复、AI 增强的总地基。
- 但它比“补一致性 / 补日志 / 补恢复”更重，短期收益没前面那么直接。
- 适合在第一梯队收口后启动。

### 7. 完善 CLI 参数体系

**优先级：中高**  
**性价比：高**

仍需增强：

- 命令行直接覆写配置项。
- 更完整的批处理参数。
- 更清晰的 `--output` / `--config` / `--mode` / `--concurrency` / `--no-cleanup` 等参数体系。

为什么性价比高：

- 开发成本不算特别高。
- 对自动化脚本、批处理、后续 API 化都很友好。
- 能明显改善非 launcher 场景下的可用性。

### 8. 提升并发治理能力

**优先级：中高**  
**性价比：中**

当前仍未完成的增强项：

- 动态降并发。
- 基于限流 / 错误率的自适应调节。
- 更细粒度的任务调度统计。

为什么不更靠前：

- 当前已经有第一版并发能力。
- 它的收益更多来自大规模任务和复杂 API 环境。
- 没有日志、恢复和批量统计打底时，贸然做高级并发治理容易调不明白。

### 9. 扩展低成本新输出格式

**优先级：中**  
**性价比：中高**

优先可考虑：

- HTML
- TXT

后续再看：

- LaTeX
- EPUB
- Excel

为什么拆开看：

- HTML / TXT 对 AST 依赖清晰、实现相对轻。
- LaTeX / EPUB / Excel 各自复杂度更高，应该延后。
- 所以“多格式输出”不能一锅端推进，得按难度拆开。

---

## 第三梯队：中优先级，价值明确但成本更高

这部分已经开始偏“能力扩展”了，适合在核心链路稳住、AST 初步成型后再做。

### 10. OCR 混合模式

**优先级：中**  
**性价比：中**

仍需实现：

- OCR 与视觉模型混合处理策略。
- 针对扫描 PDF、图片型 PDF 的专门路径。
- 根据页面类型选择 OCR / 视觉模型 / 高 DPI 模式。

可选方向：

- PaddleOCR
- Tesseract
- 云 OCR
- 模型自带 OCR 能力

为什么不更早：

- 方向很值钱，但接入后测试、调参、质量评估都会明显变重。
- 如果当前恢复、日志、AST 都没打好，OCR 只会让系统复杂度突然上升。

### 11. 转换质量可视化报告

**优先级：中**  
**性价比：中高**

仍需实现：

- 页级质量明细。
- 质量报告文件输出。
- 疑似异常页标记。
- Word 输出质量检查。
- 更适合人工复核的可视化结果。

为什么值得做但不抢第一：

- 它很适合提升人工复核效率。
- 但前提是先有更稳定的日志和任务统计，否则可视化也只是展示不稳定数据。

### 12. 自动修复能力

**优先级：中**  
**性价比：中**

仍可继续扩展：

- Markdown 表格修复。
- 公式闭合修复。
- 标题层级修复。
- 列表缩进修复。
- 乱码检测与修复。
- 模型错误文本清理。

为什么在第三梯队：

- 很实用。
- 但依赖更清晰的中间表示和质量判断依据。
- 否则容易写成一堆互相打架的修补规则。

### 13. 专业场景模板

**优先级：中低**  
**性价比：中**

仍可扩展为正式能力：

- 学术论文模式。
- 法律合同模式。
- 财务报表模式。
- 技术文档模式。
- 学习资料模式。

为什么先不急：

- 这是在主能力稳定后做“纵向深化”。
- 如果底层结构和质量控制不稳，模板越多维护越难。

### 14. AI 增强能力

**优先级：中低**  
**性价比：中低**

仍未开始的方向包括：

- 自动摘要。
- 文档问答。
- 智能分类。
- 文档对比。
- 重点信息提取。

为什么排后：

- 这些功能用户感知强，但对当前“转换链路”不是刚需。
- 成本不低，而且容易把项目重心从“转换正确性”带偏。

---

## 第四梯队：低优先级，高成本产品化方向

这些方向不是不能做，而是当前阶段性价比偏低，应该排在核心链路、架构复用和能力扩展之后。

### 15. Web UI

**优先级：低**  
**性价比：中低**

仍需实现：

- 拖拽上传。
- 实时进度。
- 转换预览。
- 历史记录。
- 失败页重试。
- 多格式下载。

为什么不先做：

- UI 会放大后端所有未解决的问题。
- 当前先把批量、恢复、日志、CLI 体系做好，后面做 UI 才不会变成漂亮壳子。

### 16. REST API 服务

**优先级：低**  
**性价比：中低**

仍需实现：

- 创建转换任务。
- 查询任务状态。
- 下载结果。
- Webhook 回调。
- 历史任务查询。

为什么后置：

- API 化对任务系统、状态管理、错误恢复要求更高。
- 现在直接上 API，很多内部能力还得补第二遍。

### 17. 本地模型支持

**优先级：低**  
**性价比：中低**

仍需打通：

- Ollama
- LocalAI
- LLaVA
- Qwen-VL 本地服务

适用场景：

- 隐私场景。
- 企业内网。
- 长期降低 API 成本。

为什么先不排前面：

- 方向有价值，但接入差异大、兼容成本高。
- 更适合作为后续平台化能力，而不是当前主线。

### 18. 文件夹监听 / 浏览器插件

**优先级：低**  
**性价比：低到中**

仍可扩展：

- 文件夹监听自动转换。
- 浏览器 PDF 一键发送转换。
- 与本地或远程服务联动。

为什么靠后：

- 这些属于入口扩展，不是核心能力增强。
- 核心链路没做扎实时，入口越多维护越累。

---

## 第五梯队：远期商业化方向

这部分先保留方向感，但不建议近期投入主开发资源。

### 19. 桌面增强版

**优先级：低**  
**性价比：中低**

仍可扩展：

- 历史记录。
- 模板系统。
- 隐私模式。
- 更完整的批量转换能力。

### 20. SaaS

**优先级：低**  
**性价比：低**

仍可探索：

- 按页数收费。
- 按并发数收费。
- 按文件大小收费。
- 按高级分析能力收费。

### 21. 企业私有化

**优先级：低**  
**性价比：低**

仍需建设：

- 内网部署。
- 权限管理。
- 审计日志。
- 数据脱敏。
- 批量 API。

### 22. API 计费

**优先级：低**  
**性价比：低**

仍可扩展：

- 按页计费。
- 按任务计费。
- 按模型成本分层计费。
- 高并发套餐。

---

## 六、重新排序后的推荐执行顺序

### A. 立刻做：低成本高收益

1. 统一配置项与真实行为。
2. 补齐边界测试。
3. 增强失败恢复与重试体验。
4. 增强结构化日志与 debug 能力。
5. 增强批量处理能力。

### B. 第二批做：搭通用骨架

6. 设计统一 AST / IR。
7. 完善 CLI 参数体系。
8. 提升并发治理能力。
9. 扩展 HTML / TXT 等低成本输出格式。

### C. 第三批做：能力增强

10. OCR 混合模式。
11. 转换质量可视化报告。
12. 自动修复能力。
13. 专业场景模板。
14. AI 增强能力。

### D. 最后做：产品化与商业化

15. Web UI。
16. REST API。
17. 本地模型支持。
18. 文件夹监听 / 浏览器插件。
19. 桌面增强版。
20. SaaS / 企业私有化 / API 计费。

---

## 七、当前最值得优先推进的 10 个任务

1. 统一 `output.add_page_separator` 等配置开关与主流程行为。
2. 补齐配置 / 缓存 / 并发 / 重试相关边界测试。
3. 增加失败页重试入口和恢复指引。
4. 增加结构化日志与请求耗时统计。
5. 做批量任务结果汇总、失败清单和跳过已完成文件。
6. 设计统一 Page / Element AST。
7. 完善 CLI 参数体系。
8. 扩展 HTML / TXT 输出。
9. 引入 OCR 混合模式。
10. 再推进 Web/API/本地模型等产品化能力。

一句话总结：**最划算的路线不是马上做大功能，而是先把现有链路补齐、测稳、能恢复，再搭 AST 这个通用底座，最后再做 OCR、Web/API 和商业化。**
